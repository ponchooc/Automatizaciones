DOCUMENTACION COMPLETA DEL SCRIPT automatizador_transfere.py

1) PROPOSITO GENERAL
Este script automatiza la validacion diaria de transferencias entre GNX y WMS.
Lee un archivo de match, intenta resolver transferencias por cache historica, valida por API en lotes, ejecuta reproceso remoto cuando hay pendientes, realiza revalidaciones temporizadas y cierra con reporte y correo.

2) ENFOQUE OPERATIVO
El flujo esta diseñado para continuidad diaria.
Si el proceso se corta, puede reanudar usando:
- estado consolidado del dia
- checkpoints de fases
- marcas de ssh y correo
- archivo de entrada archivado del mismo dia

3) ARCHIVOS PRINCIPALES
- entrada: match_13_14.txt
- control: control_transfere_ddmmyyyy.json
- txt operativo: transfere_procesado_ddmmyyyy.txt
- errores: errores_transfere_ddmmyyyy.txt
- excel: reporte_transfere_ddmmyyyy.xlsx
- log: log_transfere_ddmmyyyy.txt

4) DEPENDENCIAS EXTERNAS
- FTP GNX para descarga y subida de i04.txt.
- SSH GNX para ejecutar automatizador_transferencias.sh.
- API WMS order_hdr para validar order_nbr.
- Outlook COM para correo de cierre.

5) CARGA DE CREDENCIALES
obtener_credenciales
- Lee .acceso_wms y .acceso_gnx.
- Sin ambas credenciales el main no avanza.

6) FASE 0 FTP
fase_cero_ftp
- Conecta a ruta ORACLE en GNX.
- Busca match_13_14.txt.
- Valida que sea del dia.
- Descarga archivo local.
- Devuelve True en descarga valida.

7) CARGA DE INSUMO Y AUTORRECUPERACION DE ARCHIVO
ejecutar_verificacion
- Si no existe match_13_14.txt intenta usar match_13_14_ddmmyyyy.txt.
- Esto permite retomar proceso aun cuando el archivo original ya fue archivado.

8) PARSEO DE ENTRADA
Dentro de ejecutar_verificacion
- Cada linea puede traer:
  - transferencia
  - bodega origen
- Limpia sufijo .0 en ambos campos.
- Guarda mapa maestro:
  transferencia -> {sales_check, bodega}
- Detecta duplicados y los reporta.

9) RENOMBRADO DE CONTROL
Si el archivo activo es el original, se renombra a version historica del dia despues de cargar memoria.
Objetivo:
- evitar reprocesos accidentales del mismo archivo.
- mantener trazabilidad diaria.

10) ESTADO Y CACHE
cargar_estado
- Lee control diario NDJSON y conserva ultimo estado por transferencia.

cargar_historial_ayer
- Carga validaciones positivas del dia anterior.
- Soporta normalizacion 11 -> 12 digitos para llave de cache.

11) GUARDADO CONTINUO
hilo_guardado_continuo
- Consume cola global.
- Escribe control json y txt operativo con lock.
- Reduce riesgo de corrupcion en escritura concurrente.

12) MODELO DE VALIDACION API
consultar_lotes_ordenes
- Entrada: lista (transferencia, sales, bodega_orig).
- Consulta endpoint order_hdr por order_nbr__in.
- Si existe en API -> OK VERIFICADO API.
- Si no existe o falla -> NO MAL ERROR.
- Conserva bodega de origen para contexto de reporte.

13) FASE 1 VALIDACION MASIVA
En ejecutar_verificacion
- Separa pendientes no procesados hoy.
- Clasifica cada pendiente:
  - resuelto por cache
  - enviado a API
- Construye lotes de 100.
- Usa Session con Retry y ThreadPool de 5 workers.
- Reporta avance de lotes en consola.

14) PAUSA DE SEGURIDAD POR VOLUMEN
Durante fase 1:
- Cada 200 lotes aplica pausa de 60 segundos.
- Objetivo: aliviar presion sostenida en sesiones largas.

15) VERIFICACION DE REENTRADA
verificar_reentrada
- Reconsulta muestra corta de registros recientes no cacheados.
- Actualiza si hay cambio de etiqueta final.

16) FASE 2 REVALIDACION DE ERRORES
procesar_lote_errores
- Toma del estado consolidado solo registros NO MAL ERROR.
- Reconsulta por lotes de 100.
- Regenera estado y reporte de fase.

17) REPROCESO REMOTO GNX
subir_ftp_y_ejecutar_ssh
- Crea i04.txt con sales_check pendientes.
- Sube i04.txt a ruta GNX de consumo.
- Renombra localmente a i04_ddmmyyyy.txt.
- Inicia shell ssh y ejecuta automatizador_transferencias.sh.
- Cierra sesion de forma controlada.

18) FASE 3 Y FASE 4 CON CHECKPOINT
esperar_y_revalidar
- Parametros: sesion, fase, errores, minutos.
- Crea checkpoint .faseN_iniciada_transfere_ddmmyyyy.
- Si checkpoint existe, salta espera para recuperacion rapida.
- Reconsulta remanentes y calcula recuperadas vs fallidas.
- Genera reporte por fase.

19) REPORTE FINAL
generar_reporte_final
- Exporta excel de consolidado.
- Escribe txt de errores por fase.
- Contrasta conteo maestro con conteo de lineas para consistencia.
- Si no hay errores o es fase 4:
  - archiva archivo de entrada
  - archiva i04 si existe

20) CORREO FINAL
enviar_correo
- Evita duplicado con semaforo .correo_enviado_transfere_ddmmyyyy.
- Construye correo html con resumen ejecutivo.
- Incluye tabla de pendientes con transferencia y bodega.
- Adjunta log.
- Reintenta hasta 3 veces.

21) RUTAS EN MAIN
Ruta A proceso ya cerrado
- Si existe semaforo de correo, no hace nada adicional.

Ruta B recuperacion prioritaria
- Si hay estado previo del dia, primero ejecuta verificacion para completar pendientes de fase 1.
- Luego decide subcasos:
  - si ya hubo ssh o no hay remanentes -> revalidaciones pendientes y correo.
  - si no hubo ssh y hay remanentes -> ejecuta ssh, luego fase 3/4 y correo.

Ruta C inicio normal
- Si no hay estado previo, intenta fase 0 ftp si falta entrada.
- Si consigue entrada, ejecuta flujo completo.

22) SEMAFOROS DE CONTROL
- .chk_ssh_transfere_ddmmyyyy: evita repetir reproceso ssh diario.
- .correo_enviado_transfere_ddmmyyyy: evita reenvio de correo.
- .fase3_iniciada_transfere_ddmmyyyy y .fase4_iniciada_transfere_ddmmyyyy: evitan repetir tiempos de espera.

23) RESILIENCIA Y CONTINUIDAD
El script esta preparado para:
- relanzarse varias veces el mismo dia
- retomar desde estado consolidado
- no perder trazabilidad
- cerrar notificacion aunque haya habido corte intermedio

24) SALIDAS DE AUDITORIA
- control diario NDJSON como fuente de verdad incremental
- reporte excel para consumo de negocio
- txt de errores por fase
- log completo con cronologia de cada decision y evento

25) RESUMEN FINAL
automatizador_transfere.py implementa un pipeline diario robusto con control de estado, reintentos y recuperacion automatica.
La logica central es validar en API, escalar a reproceso remoto cuando sea necesario, y cerrar con evidencia completa para operacion y soporte.
